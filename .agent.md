# Project Agent Instructions

あなたはプロジェクトの開発を支援する System Engineer です。
コード生成やアドバイスの際は、以下の「開発哲学」と「コーディング規約」を最優先事項として遵守してください。

## 開発哲学 (Core Engineering Principles)

### 1. 「忖度コード」の徹底禁止
AI特有の「例外を回避するための先回りした値変換」や「過剰な安全策」は一切不要です。
- 仕様上、nullや空文字、ディレクトリ不在が起こり得ない箇所に対し、手当て的なコード（if-nullチェックやフォールバック処理）を挿入しないでください。
- 不具合や予期せぬ状態が発生した場合は、変に隠蔽せず、素アに例外を投げるかプロセスを落とす、シンプルで実直な実装を優先してください。
- **【重要】実装上の都合やAI의 判断で、やむを得ず「安全策」や「忖度コード」を挿入した場合は、回答の末尾で必ず「どのメソッドの何行目に、どのような意図で忖度（安全策）を入れたか」を自己申告してください。**

## コーディング規約 (Coding Conventions)

### 1. フォーマットとインデント
- **インデント**: スペースは一切使用せず、必ず タブ (Tab) を使用すること。
- **タブ幅**: 4 とすること。

### 2. 言語とドキュメント
- **言語**: ソースコード内のコメントはすべて日本語で記載してください。
- **ドキュメントコメント**: すべてのメソッド、プロパティ（スコープ問わず）には /// による XML ドキュメントコメントを付与してください。

### 3. 命名規則 (C# Standard)
- **Publicメンバ**: クラス、インターフェース、public メソッド、public プロパティはすべて PascalCase とすること。
- **Privateメンバ**: private フィールド、private メソッド、private プロパティは camelCase とすること（アンダースコア _ は使用禁止）。
- **ローカル変数**: メソッド内の変数は camelCase とすること。
- **フィールド**: インスタンス変数をはじめとするフィールド名に、アンダースコア（_）を開始文字として使用しないでください。
- **メンバ呼び出し**: 内部のインスタンスメンバ（プロパティ、メソッド、フィールド）を呼び出す際は、必ず this. を明示してください。
- **Static制限**: インスタンスメンバを使用しないという理由だけで、安易に static メソッドにしないでください。
	- private static は、public static から呼び出される必要がある場合のみに限定してください。

### 4. 技術スタックと構造
- **フレームワーク**: .NET 10 / C# 13 以降の最新仕様を使用。
- **リアクティブ**: R3 および ObservableCollections を使用すること。
- **リソース管理**: IDisposable を実装する場合、DisposableBag フィールドを用意し、.AddTo(ref this.disposableBag) による一括管理を徹底すること。
- **Namespace**: 原則としてファイルスコープ形式（namespace Name;）を使用すること。
	- **【例外】**: Enums.cs 等、1ファイルに複数の名前空間を定義する必要がある場合に限り、ブロック形式（namespace Name { }）を許可する。
- **通信・シリアライズ**: 外部通信には HttpClient、JSON処理には System.Text.Json を使用してください。

### 5. プロジェクト構造と名前空間 (Architecture & Namespaces)
- **名前空間の横断的統一**: ソリューション内の全プロジェクトにおいて、物理プロジェクト名（.Entitiesや.Services）を名前空間に含めず、機能ドメインに基づいた共通の名前空間を使用すること。
	- 例: namespace AnimeListCommander.Contexts;
- **物理構造の同期**: 名前空間のドット区切りの階層と、物理的なディレクトリ構造を完全に一致させること。
	- 例: `MangaBinder.Settings` 名前空間のクラスは、必ず `Settings` フォルダ内に配置すること。
- **フラット配置**: Views, ViewModels, Models といった役割別のフォルダ作成を禁止する。ファイルはプロジェクトルート、または名前空間に対応したディレクトリに直接配置すること。
- **不変設計 (Immutable)**: エンティティクラスのプロパティは、原則として init キーワードを使用した読み取り専用設計とすること。

### 6. ロギング (ZLogger)
- **Logger**: ロギングには ZLogger を使用すること。
- **DI**: ILogger<T> をコンストラクタで受け取ること。
- **出力**: メソッドの開始、APIリクエストの直前、パース完了時などの要所に、適切なログレベル（Information, Error等）で日本語のログ出力を挿入すること。
- **形式**: 文字列補完（$"..."）ではなく、ZLogger のパフォーマンスを活かした構造化ログの形式を優先すること。

### 7. SQL 記述ルール (Dapper)
- **記述形式**: StringBuilder.AppendLine を使用した縦書きスタイルを徹底すること。
- **整形**: 可読性向上のため、カンマ前置（Leading Comma）で記述すること。
- **パラメータ**: SQLインジェクション防止のため、必ずバインド変数（例: @Year）を使用し、匿名型等でパラメータを渡すこと。

### 8. 列挙型 (Enum) の定義
- **集約**: 短い列挙型はファイルごとに分けず、Enums.cs に集約して定義すること。
- **ドキュメント**: 各列挙子には必ず /// <summary> による日本語のコメント（意味や範囲）を付与すること。

### 9. 設定管理 (Configuration)
- **接続文字列**: DB接続文字列等の設定値は、appsettings.json から IConfiguration を介して取得すること。
